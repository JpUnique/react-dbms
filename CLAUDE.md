# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Install dependencies (pnpm is required — do not use npm or yarn)
pnpm install

# Start dev server (http://localhost:3000)
pnpm dev

# Production build (outputs to dist/)
pnpm build

# Lint
pnpm lint

# Preview production build locally
pnpm preview
```

## Environment

One variable is needed:

```
VITE_API_URL=http://localhost:4000/dbms/v1
```

If absent, `src/services/api.ts` falls back to `http://localhost:4000/dbms/v1`. The dev server binds to `0.0.0.0:3000` and expects CORS from the Go backend (which allows `http://localhost:3000`).

## Architecture

### Provider nesting (App.tsx)

The provider tree wraps the router, in this order outermost → innermost:

```
QueryClientProvider → LicenseProvider → AuthProvider → BrowserRouter
```

`LicenseProvider` is deliberately outside `AuthProvider` — license validation happens before and independently of authentication.

### Data layer

All pages and components use the **service layer** (`src/services/*.service.ts`) — plain async functions that call the Go backend via HTTP. Import via `@/services` (barrel export at `src/services/index.ts`).

`src/context/DocumentContext.tsx`, `ActivityContext`, `UserManagementContext`, and `ShareContext` still exist in `src/context/` but are **no longer mounted in the provider tree** — they are dead code and should not be imported by new or migrated code.

### Service layer

`src/services/api.ts` is the central HTTP client. It:
- Reads the JWT access token from `localStorage` under the key `"token"`
- Sets `Authorization: Bearer <token>` on every request automatically
- Throws `Error` with the `error` field from the backend response body on non-2xx

All service modules export plain object namespaces (e.g. `authService`, `documentsService`, `foldersService`). Import them via `@/services`.

### Auth flow

- `AuthContext` calls `authService.getCurrentUser()` on mount, which hits `GET /auth/me` using the stored token
- On login, the access token is stored via `auth.setToken()` (`localStorage` key `"token"`); the refresh token is not currently persisted on the frontend
- `ProtectedRoute` and `PublicRoute` in `App.tsx` guard routes by reading `currentUser` from `AuthContext`

### License system

The license system is entirely client-side and `localStorage`-backed — there is no backend license endpoint:
- `src/lib/licenseManager.ts` generates, stores, and validates license keys
- Keys are stored in `localStorage` under `docmanager_license` (active license) and `docmanager_licenses_db` (admin-side license database)
- `LicenseContext` validates on mount and rechecks every hour
- `/activate-license` and `/admin/licenses` are the user-facing license routes

### Layout

All authenticated pages use `MainLayout` (`src/components/layout/MainLayout.tsx`), which renders `Sidebar` + `TopBar` + a scrollable `<main>`. Pages import and wrap their content in `MainLayout` themselves.

### Path alias

`@/` resolves to `src/`. All internal imports should use this alias.

### Branding

All visible product name, logo, and color references come from `src/config/branding.ts`. Editing that file is the only thing needed to rebrand the app.

### UI components

`src/components/ui/` contains shadcn/ui components (Radix-based). Do not edit these directly — they are generated by the shadcn CLI. Add new shadcn components with:

```bash
pnpm dlx shadcn@latest add <component-name>
```

Custom app components live in `src/components/layout/` and `src/components/document/`.
